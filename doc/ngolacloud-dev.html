Workstation provisioning · Kind · Cilium · GitOps · SLSA L3 · 14 tiers

Quickstart — 30 minutos

O caminho feliz para um novo dev. Cada comando é idempotente.

git clone https://github.com/angolardevops/ngolacloud-dev-setup
cd ngolacloud-dev-setup

make validate                       # preflight (sem sudo, sem writes)
make onboard                        # one-shot ~15 min
make health                         # 8 ✓ esperados
xdg-open http://localhost:3000      # Grafana — admin / ngolacloud-dev

Para passo-a-passo manual:

make setup-check                    # ansible dry-run
make setup                          # apply ~10 min
make reboot-if-needed
make kind-up WITH_OBS=1              # cluster + Cilium + Grafana

PR workflow + Gitflow

Modelo: trunk-based + short-lived feature branches. Sem gitflow clássico (release/develop) — overhead desnecessário para esta equipa.

Branch naming

feat/           # nova capability (T11.1)
fix/            # bug fix
chore/          # housekeeping
docs/           # docs only
refactor/       # no behaviour change
ci/             # workflow changes

PR checklist (cada PR)

  1. Branch criada com nome semântico
  2. Commits seguem Conventional Commits (ver próxima secção)
  3. make lint passa localmente (pre-commit hook ajuda)
  4. make health verde antes do push
  5. CHANGELOG.md actualizado se for user-facing
  6. ADR escrito se a mudança tem implicação arquitectural
  7. Tests adicionados ou actualizados
  8. PR description: contexto / decisão / trade-offs (não apenas "what")
  9. Review de pelo menos 1 outro engenheiro antes do merge
  10. Squash + merge (mantém main linear)

pip install pre-commit

SOLID + CQRS + Clean Arch

Princípios aplicáveis quer no Python (portal), quer no Rust (CLI), quer nos Ansible roles:

SOLID

PrincípioAplicação NgolaCloud
Single ResponsibilityCada Ansible role faz uma coisa. system_tuning não toca em Docker.
Open/ClosedTiers são aditivos. Tier 11 não modifica Tier 9; estende.
Liskov SubstitutionO portal-chart reference é substituível pelo chart real do portal (mesmo contrato).
Interface SegregationScripts têm 1 propósito + flags claros. kind-up.sh não faz kind-down.
Dependency InversionO CLI depende de .ngolacloud.toml (interface), não de Hetzner-specific code.

CQRS — Command/Query Responsibility Segregation

No portal Django: endpoints de leitura (GET /api/v1/admin/overview/) consultam um aggregator read-only; endpoints de escrita (POST /api/v1/instances/start/) publicam eventos que actualizam o estado. Não misturas READ + WRITE no mesmo serializer.

# Read side — apps/api/v1/admin_screens.py
class AdminOverviewView(APIView):
    def get(self, request):
        return Response({"stats": aggregate_stats(), ...})

# Write side — apps/instances/services.py
class InstanceService:
    def start(self, instance_id):
        emit_event("instance.start.requested", instance_id)
        return AcceptedResponse()  # 202, estado eventual

Clean Architecture — concentric layers

  • Entities (models.py) — Tenant, Instance. Pure data + invariants.
  • Use cases (services.py) — InstanceService.start(). Orquestra entities, não conhece HTTP.
  • Adapters (views.py / serializers.py) — Traduz HTTP ↔ use cases.
  • Frameworks (Django, DRF) — Detalhe externo. Substituível.

Regra: dependências apontam para dentro. Entities não importam Django.

SRE essentials

  • SLO/SLI — define target uptime (99.9% prod) e error budget (43min/mês). Mede via Prom.
  • Runbooks — cada alerta tem um runbook URL. docs/troubleshooting.md + futuras runbooks/.
  • Blameless postmortems — incidents geram um postmortem em docs/postmortems/YYYY-MM-DD-titulo.md.
  • Chaos engineering — Tier 11 instala chaos-mesh; experimenta em staging primeiro.
  • Capacity planning — opencost projecta custos; HPA + cluster-autoscaler em prod.

Troubleshooting rápido

"missing helm — run 'make setup' first"
Phase 3 do onboard.sh foi declinada. Corre make setup TAGS=kind (~2min) ou instala helm manual via curl.
"cluster ngolacloud-dev already exists"
Cluster antigo presente. make kind-up-recreate destrói + cria com a config nova (perde dados em volumes), ou make kind-down && make kind-up.
Resource Slice not configured
make setup TAGS=slice resolve em ~5s. Necessário para que Docker respeite o limite de 32 GB.
Disco a > 90%
make prune (safe) ou make prune-aggressive (também volumes). make kind-down liberta ~10-20 GB.
PATH_add: command not found
Estás a fazer source .envrc sem direnv. O ficheiro tem fallback shims na v1.1+; faz git pull.

Lista completa em docs/troubleshooting.md.
